# Bilibili Login Data Security Analysis

## Overview

Bilibili login information (Cookies) is encrypted using AES256-GCM and stored locally, with keys saved in Android KeyStore.

Security features:
- Login information encrypted, inaccessible by other apps
- Data automatically destroyed upon app uninstallation
- Not uploaded to any servers, only saved on local device

## Technical Implementation

### 1. Encrypted Storage Solution

Using EncryptedSharedPreferences:
- Encryption algorithm: AES256-GCM
- Key storage: Android KeyStore
- Key scheme: AES256_SIV (key encryption) + AES256_GCM (value encryption)

### 2. Stored Data Content

- Cookie: Contains user session information (SESSDATA, bili_jct, DedeUserID, etc.)
- User info: UID, username, avatar link, etc.
- Refresh token: For maintaining long-term login state

### 3. Security Features

Implemented protection measures:

1. Data encryption: All Cookies and user information encrypted with AES256
2. Hardware protection: Keys stored in Android KeyStore, protected at hardware level
3. App isolation: Android sandbox ensures other apps cannot access encrypted data
4. Tamper-proof: GCM mode provides authenticated encryption, detects and rejects tampered data

## Potential Risks

### 1. Rooted Device Risk

Users with Root privileges can theoretically attempt to access KeyStore. Recommended to avoid using sensitive accounts on rooted devices, or regularly log out.

### 2. App Reverse Engineering

Attackers may decompile the app to analyze code logic. App uses ProGuard code obfuscation to increase reverse engineering difficulty. Even knowing encryption method, cannot decrypt without the key.

### 3. Memory Attacks

When app is running, data briefly exists in memory (plaintext). Requires Root privileges and professional memory debugging tools to attack, and can only obtain current runtime session.

## Implemented Security Measures

- Use EncryptedSharedPreferences for encrypted storage
- Enable ProGuard code obfuscation
- Provide logout functionality
- Automatically clear all data upon app uninstallation

## Security Recommendations

Basic security practices:
1. Regular logout: Recommended to log out account when not in use for extended periods
2. Avoid rooted devices: Rooting reduces overall security
3. Keep app updated: Install latest version for security fixes
4. Enable screen lock: Set up password/fingerprint lock on phone

## Technical Implementation Details

### EncryptedSharedPreferences Working Principle

```
User login
   ↓
Get Cookie (plaintext)
   ↓
AES256-GCM encryption ← Android KeyStore provides key
   ↓
Encrypted data stored on phone
   ↓
When needed
   ↓
Get key from KeyStore → Decrypt → Use (only in memory)
```

### Key Management Mechanism

Key generation:
```kotlin
val masterKey = MasterKey.Builder(context)
    .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
    .build()
```

Features:
- Key automatically generated by Android system
- Key never leaves Android KeyStore
- Key automatically destroyed upon app uninstallation
- Reinstalling app generates new key, old data cannot be decrypted

### Why Choose This Solution

Compared to self-implemented encryption:
- Easy to have security vulnerabilities (e.g., hardcoded keys)
- Difficult key management
- Easy to be reverse engineered

Using Android official solution:
- Maintained by Google security team
- Validated by numerous real applications
- Automatically handles key management
- Continuous updates and vulnerability fixes

## Summary

This app uses EncryptedSharedPreferences encryption solution, suitable for video player scenarios:
- Protects user privacy
- Complies with Android security best practices
- Prevents common attack methods
- Minimal performance overhead
